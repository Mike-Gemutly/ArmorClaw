## QR Code Contents

### Payload Structure

The QR code contains a JSON payload with verification data:

```json
{
  "v": 1,
  "type": "device_verification",
  "device_id": "ABCD1234EFGH",
  "public_key": "ed25519:AbCdEf123456789...",
  "emojis": ["ğŸ¦", "ğŸš€", "ğŸ¸", "ğŸŒŠ"],
  "expires_at": 1739667600,
  "server": "https://matrix.example.com",
  "user_id": "@user:matrix.example.com"
}
```

### Payload Fields

| Field | Type | Purpose |
|-------|------|---------|
| `v` | int | Protocol version |
| `type` | string | Always "device_verification" |
| `device_id` | string | Unique device identifier |
| `public_key` | string | Device public key for verification |
| `emojis` | array | 4 emoji for visual verification |
| `expires_at` | int | Unix timestamp (5 min expiry) |
| `server` | string | Matrix homeserver URL |
| `user_id` | string | User's Matrix ID |

### QR Code Format

- **Type:** QR Code (Model 2)
- **Encoding:** Binary
- **Error Correction:** Level M (15% recovery)
- **Size:** Variable (typically 21x21 to 25x25)

## Security Considerations

### Code Security

1. **Time-limited**: Codes expire after 5 minutes
2. **Single-use**: Each code can only verify one device
3. **Random**: Verification codes are cryptographically random
4. **Encrypted**: QR payload is encrypted in transit

### Emoji Verification

The emoji comparison provides a visual verification that:
- Prevents MITM attacks on the QR code channel
- Ensures both parties see the same verification data
- Is human-readable and easy to compare

### Best Practices

1. Always verify emoji match before confirming
2. Generate new code if expiry is approaching
3. Use secure network for verification
4. Revoke verification if mismatch detected
